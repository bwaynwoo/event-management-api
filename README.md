# Event Management API

REST API для управления мероприятиями.
## Технологии

- .NET 10
- ASP.NET Core Web API
- Swagger/OpenAPI
- In-memory хранилище

## Требования

- [.NET 10](https://dotnet.microsoft.com/download) или выше

## Установка и запуск

* Клонировать репозиторий

```bash
git clone https://github.com/bwaynwoo/event-management-api.git
```

* Перейти в репозиторий
```bash
cd event-management-api
```

* Переключиться на ветку sprint-2

```bash
git switch sprint-2
```

* Собрать решение
```bash
dotnet build
```

* Запустить проект
```bash
dotnet run
```

* Открыть Swagger\
После запуска Swagger будет доступен по адресу:
```
http://localhost:5000/swagger/index.html
```

>Порт может отличаться. Точный адрес будет показан в терминале после запуска.

## API Endpoints

| Метод   |	Endpoint	| Описание |
|---------| ------------ | -------- |
| GET	    |/events	|Получить список всех мероприятий|
| GET	    |/events/{id}	|Получить мероприятие по ID
| POST	   |/events	|Создать новое мероприятие
| PUT	    |/events/{id}	|Обновить мероприятие целиком|
| DELETE	 |/events/{id}	|Удалить мероприятие|
| POST    |/events/{id}/book |Cоздание брони по указанному id события|
| GET     |/bookings/{id} | Получение брони по id |

### Особенности реализации
* Данные хранятся в памяти приложения (ConcurrentDictionary<T>)
* При перезапуске сервера все данные сбрасываются
* ID генерируется автоматически
* Swagger доступен в режиме разработки

## Параметры фильтрации для GET /events

| Параметр | Тип | Обязательный | Описание |
|----------|-----|--------------|----------|
| title | string | Нет | Поиск по названию (регистронезависимый, частичное совпадение) |
| from | DateTime | Нет | События, которые начинаются не раньше указанной даты |
| to | DateTime | Нет | События, которые заканчиваются не позже указанной даты |

## Формат ответа при ошибках

При возникновении ошибок API возвращает JSON-ответ в следующем формате:

```json
{
  "status": <код_статуса>,
  "detail": "<сообщение_об_ошибке>"
}
```

### HTTP статусы:
| Статус |Описание|
|--------|--|
| 400    |	Ошибка валидации (некорректные данные запроса)
| 404    |	Ресурс не найден (событие с указанным ID отсутствует)
| 500    |	Внутренняя ошибка сервера

### Модель Booking
| Поле|Описание|
|-------------------------------|--|
| Id (Guid, обязательное)       | уникальный идентификатор брони|
| EventId (Guid, обязательное) | идентификатор события, к которому относится бронь| 
| Status (BookingStatus, обязательное) | текущий статус брони|
|CreatedAt (DateTime, обязательное) | дата и время создания брони|
|ProcessedAt (DateTime?, опциональное) | дата и время обработки брони|

#### BookingStatus

| Поле|Описание|
|-----|--|
|Pending | бронь создана, ожидает обработки|
|Confirmed | бронь подтверждена|
|Rejected | бронь отклонена|

### Фоновый сервис
Реализован фоновый сервис, который периодически опрашивает хранилище на наличие бронирований в статусе Pending.
Для каждой необработанной брони выполняется искусственная задержка (например, Task.Delay на 2 секунды), имитирующая обращение к внешней системе;
После задержки бронь переводится в статус Confirmed.
После смены статуса заполняется поле ProcessedAt.
Обновлённая бронь сохраняется в хранилище.

### Пример сценария использования
1. Создаём событие
2. По id события создаём бронирование
3. Убеждаемся что бронирование создано и в статусе - "Confirmed"
